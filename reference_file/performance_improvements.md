## 성능 개선 기록

### 1차 개선: first-fit → next-fit + realloc 제자리 확장

**66점 → 86점 (+20점)**

#### 처리량 (thru): 21/40 → 40/40 (만점)

```
first-fit:   318 Kops
next-fit:   1865 Kops  ← AVG_LIBC_THRUPUT(600K) 초과 → 만점
```

next-fit 하나로 처리량 5.8배 향상. 매번 `heap_listp`부터 순회하던 것을
마지막 탐색 위치(`rover`)부터 이어서 탐색하도록 변경.

#### 공간 활용도 (util): 74% → 76%

| trace | 이전 | 현재 | 변화 | 원인 |
|-------|------|------|------|------|
| 0~3 | 99~100% | 91~97% | 하락 | next-fit은 처음 만나는 블록 할당 → 덜 최적 |
| 4 | 66% | 66% | 동일 | binary 패턴, 근본 개선 필요 |
| 5~6 | 92% | 89~91% | 소폭 하락 | next-fit 영향 |
| 7~8 | 55%, 51% | 55%, 51% | 동일 | binary 패턴, 근본 개선 필요 |
| 9 | 27% | 27% | 동일 | realloc 제자리 확장이 이 패턴엔 미효과 |
| **10** | **34%** | **82%** | **+48%** | realloc 제자리 확장 효과 |

#### 남은 문제

**① trace 7, 8 (binary 패턴) — 55%, 51%**
```
작은 블록(e.g. 16B)과 큰 블록(e.g. 4096B)을 번갈아 할당
→ 큰 free 블록이 작은 블록으로 쪼개지면서 외부 단편화 심각
→ segregated free list 없이는 개선 불가
```

**② trace 9 (realloc) — 27%**
```
인접 블록이 free가 아닌 경우 제자리 확장 불가
→ 매번 새 블록 할당 + 복사 → 힙 낭비
→ 이전 블록까지 고려하는 realloc 전략 필요
```

#### 추가로 올릴 수 있는 점수

| 개선 | 효과 |
|------|------|
| segregated free list | trace 7, 8 util 대폭 향상 → util +5~10% 예상 |
| realloc 이전 블록 방향 확장 | trace 9 util 향상 |
| best-fit (일부 구간) | trace 0~3 util 회복 |

현재 처리량은 이미 만점이므로, 이후 개선은 **util 점수(현재 46/60)** 에 집중.
